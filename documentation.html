Introduction à la Documentation

L'objectif est d'expliquer l'importance de la documentation dans le développement d'une application web et de montrer comment bien la structurer.

Pourquoi documenter ?

Une documentation bien rédigée est essentielle pour assurer la pérennité et l’évolution d’une application.
Elle permet de :

  • Communiquer :
    Si vous écrivez du code aujourd'hui et que vous revenez dessus dans six mois, vous serez très content d'avoir laissé quelques notes.
    Si vous travaillez en équipe, la documentation est ce qui assure la transmission des connaissances.
  • Faciliter la maintenance et la mise à jour :
    Un développeur reprenant un projet doit comprendre rapidement son architecture et son fonctionnement.
  • Améliorer la collaboration :
    Dans une équipe, une documentation claire permet à chacun de travailler efficacement sans perte de temps à poser des questions.
    Entre les développeurs, mais aussi avec les autres parties prenantes (chefs de projet, clients).
  • Assurer la transmission des connaissances :
    Un projet peut évoluer sur plusieurs années et plusieurs développeurs, il est crucial d’enregistrer les décisions techniques.
  • Réduire les erreurs et la dette technique :
    Une documentation précise limite les mauvaises implémentations et les failles de sécurité.

Types de documentation

On distingue plusieurs types de documentation, chacune ayant son rôle :

  1. Documentation utilisateur
    • Guides et tutoriels
    • Destinée aux utilisateurs finaux.
    • Contient des guides d’utilisation, des FAQ et des tutoriels.
    • Exemples : Pages d’aide d’un site web, documentation API publique.
  2. Documentation fonctionnelle
    • Les règles métier
    • Elle décrit ce que doit faire l'application.
  3. Documentation technique
    • Architecture, API, base de données
    • Décrypte l’architecture du projet, les choix techniques et les dépendances.
    • Cible les développeurs et les administrateurs système.
    • Contient des schémas, diagrammes et explications détaillées.
  4. Documentation du code
    • Commentaires, conventions de nommage
    • Intégrée directement dans le code source.
    • Utilise des commentaires et des outils comme JSDoc ou DocFX.
    • Explique les classes, fonctions et variables importantes.

Et bien sûr, toutes ces documentations doivent être claires, accessibles et à jour.

Bonnes pratiques de documentation

Clarté et structure

  • Soyez clair et concis.
  • Organiser les informations.
  • Utiliser un sommaire et une navigation intuitive.
  • Utilisez des titres, des tableaux et des schémas pour organiser l'information.
  • Définissez un format standard pour tout le projet.

Mise à jour régulière

  • Documentation vivante et non statique.
    Une documentation périmée est plus nuisible qu’une absence de documentation.
  • Automatiser les mises à jour via CI/CD.
    Automatiser la mise à jour avec des outils comme GitHub Actions pour générer des docs à partir du code.

Standardisation

  • Utiliser des conventions d’écriture (ex. : norme pour les commentaires de code).
  • Employer des formats adaptés :
    • Markdown pour une documentation simple et versionnée.
    • AsciiDoc pour une documentation plus complète.
    • Swagger/OpenAPI pour documenter une API REST automatiquement.
  • Exemples :
    • Normes : ISO/IEC 26514, Google Developer Documentation
    • Templates et conventions : RFC, "README.md" standardisé

Outils et technologies recommandés

Documentation versionnée et collaborative

  • Markdown, GitHub Wiki : Documentation simple, stockée avec le code source.
  • Confluence, Notion : Outils pour collaborer.
  • Read the Docs : Génération automatique d’une documentation hébergée.
  • Google Docs : Documents classiques.

Génération de documentation interactive

  • Docusaurus : Création d’une documentation web interactive avec une navigation intuitive.
  • MkDocs : Alternative légère pour documenter un projet Python.
  • VuePress : Génération d'un site à partir de Markdown.

Documentation des API

  • Swagger / OpenAPI : Génération de documentation dynamique à partir du code de l’API.
  • Postman : Outil de test d’API avec documentation interactive intégrée.

Documentation du code source

  • JSDoc (JavaScript), DocFX (.NET), Sphinx (Python), Doxygen :
    Génération automatique de documentation à partir des commentaires de code.

Cycle de vie de la documentation

Une documentation bien gérée dès ces premières étapes garantit une meilleure exécution du projet et une anticipation des problèmes.

Les 6 documents avant le développement

  1. Expression du besoin
    Le point de départ : problème à résoudre, attentes.
    Ce document recueille les attentes du client ou des utilisateurs finaux sans entrer dans les détails techniques.
    Il sert de point de départ pour comprendre les objectifs du projet.
  2. Cahier des charges fonctionnel
    Décrit les fonctionnalités attendues, les règles métier et les exigences de l'application.
  3. Cahier des charges technique
    Spécifie les choix techniques, les technologies utilisées, les contraintes d’architecture et de performance.
  4. Plan de gestion des risques
    Identifie les risques potentiels liés au projet (techniques, organisationnels, financiers) et propose des stratégies d’atténuation.
  5. Analyse des parties prenantes (RACI)
    Permet d’identifier les acteurs du projet et leurs responsabilités (qui est responsable, qui est consulté, qui est informé, etc.).
    R : responsible (on utilise aussi parfois le terme réalisateur car il réalise l'activité).
    A : accountable (on utilise aussi parfois le terme approver (approbateur), donc "validateur").
    C : consulted.
    I : informed.
  6. Plan de tests et validation
    Décrit les méthodes de test à appliquer pour valider le bon fonctionnement de l’application.

Documentation évolutive en phase de développement

  • Versionnage
  • Validation des docs via Pull Requests
  • Intégration dans les sprints Agile

Pour aller plus loin

Voici quelques aspects complémentaire à la documentation.

Documentation et qualité du Code

  • Normes et conventions : Respecter des standards comme le guide de style Airbnb pour JavaScript ou PEP 8 pour Python.
  • Linting et formatage automatique : Utilisation d’ESLint, Prettier, Black (Python) pour assurer une cohérence dans le code.
  • Tests et documentation : Générer automatiquement une documentation des tests avec Jest, Mocha, ou PHPUnit.

Documentation et accessibilité

  • Rendre la documentation inclusive :
    • Éviter les termes trop techniques sans explication.
    • Ajouter des exemples concrets et des cas d’usage.
  • Compatibilité avec les outils d’accessibilité :
    • Générer des versions PDF pour les personnes ayant des difficultés avec les formats web.
    • Vérifier la lisibilité avec des outils comme Hemingway Editor.
    • Compatibilité avec lecteurs d’écran.
    • Version audio.

Documentation et sécurité

  • Sensibilisation aux bonnes pratiques : Ajouter une section dédiée à la sécurité (ex : protection des API, gestion des clés d’accès).
  • Exemples d’attaques courantes : Injection SQL, XSS, mauvaise gestion des authentifications.
  • Documentation de la gestion des secrets : Ne jamais inclure des clés API en clair, utiliser des fichiers .env.
  • Conformité RGPD et normes de cybersécurité.

Documentation et intelligence artificielle

  • Utilisation d’IA pour améliorer la documentation :
    • ChatGPT, Copilot ou Tabnine pour aider à générer des descriptions claires.
    • Outils comme Mintlify ou Stenography pour générer automatiquement des explications à partir du code source.
  • Traduction automatique : Rendre la documentation multilingue avec DeepL API ou Google Translate API.

Documentation et collaboration

  • Utilisation d’outils collaboratifs : Notion, Confluence, Slite pour rédiger en équipe.
  • Workflow d’approbation : Faire relire la documentation via des Pull Requests Git.
  • Intégration à un CI/CD : Automatiser la génération de documentation avec des scripts (ex : déclencher la mise à jour de la documentation après un commit sur main).

Conclusion


Résumé des idées clés :

  • La documentation est une communication efficace.
  • Elle doit être claire et accessible.
  • Une documentation efficace est évolutive et collaborative.
  • Elle commence dès l’expression du besoin et suit tout le cycle de vie du projet.
  • Automatiser la documentation permet de la maintenir à jour et de gagner du temps.
